Машинный backlog: что мешает RC
P0 — блокеры до RC
Убрать архитектурную двойственность provider/gateway
Найти все execution path, где платежи идут через PaymentProviderInterface.
Найти все execution path, где платежи идут через Gateway\PaymentGatewayInterface.
Свести к одному каноническому orchestration path.
Второй path либо удалить, либо оставить только как тонкий adapter compatibility layer.
Исправить DI, handlers, controllers, tests, чтобы не осталось двух равноправных моделей исполнения.
Убрать silent failure из инфраструктуры
Найти все catch (Exception|\Throwable) с пустым телом или фактическим проглатыванием ошибки.
Заменить на явное поведение: structured logging, domain/infrastructure exception, retry mark, DLQ mark, metric increment.
Особо пройти:
outbox publisher
outbox worker
projection sync/repository
SLA/reporting
repository helpers
После правки ни один инфраструктурный сбой не должен исчезать бесследно.
Довести projection/read model до консистентности
Сравнить schema payment_projection с тем, что реально пишет projection sync.
Если поле существует в projection schema, но не заполняется, либо начать его заполнять, либо удалить из схемы.
Выровнять:
entity/repository
migration
sync service
status/metrics consumers
Добиться, чтобы read model была не “урезанным случайным срезом”, а осознанной проекцией.
Решить судьбу orderId
Проверить, где orderId приходит, валидируется, логируется, теряется.
Принять одно решение:
либо orderId — полноценная часть payment aggregate / storage / projection / API,
либо это внешний correlation key, и тогда это надо явно зафиксировать и последовательно провести через код.
Убрать текущее полусостояние, где параметр есть, но модель им не владеет полноценно.
Привести provider dependencies к честному состоянию
Проверить все реальные внешние SDK зависимости.
Если в коде есть production-провайдер, его SDK должен быть явно отражён в composer dependencies или вынесен в чётко оформленный optional adapter package.
Убрать состояние “код как бы поддерживает провайдера, но зависимость формально не собрана”.
P1 — сильно тянет вниз качество, но не абсолютный стоп
Собрать единую карту state transitions
Выписать все места, где меняется PaymentStatus.
Свести переходы в одну каноническую матрицу.
Убрать неявные или дублирующиеся переходы из разных сервисов.
Проверить, что webhook/reconciliation/manual/API flows не конфликтуют по allowed transitions.
Очистить границу synchronous flow vs async flow
Определить, какие изменения aggregate допустимы только синхронно, а какие приходят только через event/webhook/reconciliation.
Убрать дублирование одной и той же бизнес-операции в нескольких каналах.
Зафиксировать единственный authority source для статуса платежа в каждом сценарии.
Ужесточить идемпотентность
Проверить создание, старт, финализацию, refund, webhook ingest, replay.
Для каждой операции определить идемпотентный ключ и гарантии повторного вызова.
Добить тесты на повторный запрос, race, duplicate webhook, replay after partial failure.
Привести reconciliation к строгой модели
Проверить, не размыта ли ответственность между provider reconcile, webhook ingestion и local status mutation.
Сформулировать reconciliation как отдельную бизнес-операцию с чёткими входами/выходами.
Исключить скрытое “самоисцеление” без аудита причины.
Усилить observability
На каждый критический path добавить:
structured logs
metrics
correlation/payment reference
failure reason
Проверить, что любой failed payment, failed refund, failed publish, failed replay, failed projection sync объясним по логам и метрикам.
P2 — снижает промышленную зрелость
Схлопнуть transitional/legacy слои
Найти классы, которые выглядят как переходные shim/compatibility objects.
Отметить, что из них ещё используется, а что только тащит старую модель.
Удалить мёртвые и полумёртвые слои после консолидации execution path.
Нормализовать provider boundary
Внутри ядра не должно быть утечки provider-specific semantics без адаптера.
Убрать прямую зависимость бизнес-сервиса от провайдерных деталей, если такие места есть.
Привести payload normalization к единому внутреннему формату.
Проверить consistency между write-side и read-side DB topology
Убедиться, что infra DB не становится местом бизнес-истины.
Проверить watermark/lag/update semantics.
Убрать риск, когда operator/status view показывает несогласованное состояние как будто это истина.
Почистить operator console boundary
Проверить, не обходит ли console UI доменные сервисы прямыми инфраструктурными вызовами.
Вся операторская поверхность должна идти через те же application/domain-safe paths, а не через shortcut-логику.
Усилить security edge behavior
Проверить поведение при невалидном token, пустом scope, недоступном JWKS, сломанном cache.
Убедиться, что деградация идёт в deny-safe mode, а не в permissive mode.
